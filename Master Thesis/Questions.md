## Главная проблема МТС

**Chunking больших CSV:**
- Retriever возвращает 4 строки из CSV — слишком мало контекста
- Полный документ не помещается в контекст модели
- Нет промежуточного уровня: либо 4 строки, либо всё
- Нужен инструментарий для промежуточного результата

С размеченными документами (DOCX, RTF) проблем нет — индексация работает нормально.

## Что предложил ментор

- Сделать **отдельный прототип** сервиса «общение с документами» (не привязываться к LibreChat)
- Клиент перетаскивает файлы → задаёт вопросы → получает ответы
- GraphRAG можно применить для сравнительного анализа нескольких документов (10+)
- Можно обобщить тему диплома: «построение системы знаний для анализа больших объёмов данных»
- Может дать доступ к хранилищу метрик (Nginx логи, Prometheus) если понадобится

### Исследование: chunking для CSV

На что обращать внимание:
- Текущий чанкинг BGE-M3 не учитывает табличную структуру CSV
- CSV — это строки + столбцы, обычный text splitter режет по токенам, теряя структуру
- Нужен **table-aware chunking**: разбивать CSV на группы строк, сохраняя заголовки
- Промежуточный размер чанка: не 4 строки и не весь файл, а ~200-300 токенов с контекстом заголовков

Что изучить:
- [ ] Как LibreChat/RAG API сейчас делает chunking CSV (какой splitter, какой chunk size)
- [ ] Table-aware chunking strategies (row-group splitting с заголовками)
- [ ] Semantic chunking для табличных данных
- [ ] Можно ли делать pre-summarization: LLM суммаризирует CSV → суммари индексируется → retriever находит суммари → подтягивает оригинальные строки

### Прототип: сервис «общение с документами»

На что обращать внимание:
- Должен быть **максимально простым** — drag & drop файлов + текстовый ввод
- Не интегрировать в LibreChat — отдельный сервис
- GraphRAG здесь уместен: строить граф по загруженным документам, потом искать по нему
- Мультимодальность: PDF + CSV + картинки (Vision)

Архитектура прототипа:

```
Файлы → [Indexer] → Граф / Эмбеддинги
                          ↓
Запрос → [Query Processor] → [Retriever] → [Generator] → Ответ
```

Что реализовать:
- [ ] Table-aware indexer для CSV (главная боль МТС)
- [ ] GraphRAG pipeline для нескольких документов (моя тема)
- [ ] Базовые метрики: latency, retrieval recall, answer faithfulness

### Метрики — внедрить с нуля

Сейчас ничего не измеряется. Предложить минимальный набор:

| Метрика | Зачем | Как измерить |
|---------|-------|-------------|
| Retrieval recall@k | Находит ли retriever нужные чанки | Ground-truth датасет с вопросами + правильными чанками |
| Answer faithfulness | Нет ли галлюцинаций | LLM-judge или ручная разметка |
| Latency (ms) | SLA для production | Замер end-to-end времени ответа |
| Chunk coverage | Сколько релевантной информации покрыто | % ключевых фактов из документа, попавших в контекст |

Chunk coverage — специфичная метрика для проблемы CSV: показывает, насколько контекст покрывает нужную информацию.

### GraphRAG — где уместен

Ментор подтвердил: GraphRAG не нужен для текущего простого RAG. Но уместен для:
- Сравнительный анализ нескольких документов (10+ файлов)
- Построение «системы знаний» по корпусу документов
- Цепочки рассуждений по документу (Knowledge Graph of Thoughts)

Мой фокус: **прототип GraphRAG для анализа документов**, не для текущего LibreChat.

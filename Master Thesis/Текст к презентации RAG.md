## Слайд 2: Актуальность

Давайте рассмотрим актуальность проблемы. Одной из задач клиентов МТС, которые хранят корпоративные документы в облаке, является получение аналитики по ним через запросы на естественном языке. Этими документами могут быть договора, отчёты и структурированные таблицы в форматах PDF, DOCX и CSV.

Наиболее перспективным классом решений для этой задачи является RAG-инфраструктура. Однако существующие реализации не обеспечивают одновременно высокое качество аналитики и низкую вычислительную стоимость.  

---

## Слайд 3: Компоненты RAG и обзор существующих решений

Рассмотрим компоненты RAG-инфраструктуры. В этом пайплайне ответ на запрос формируется на основе фрагментов, извлечённых из индекса документов. Пайплайн включает пять компонентов. Indexer строит индекс из документов. Query Processor принимает запрос пользователя. Retriever извлекает фрагменты из индекса. Organizer формирует контекст. Generator генерирует ответ.

Для Query Processor, Organizer и Generator уже есть лучшие в своём классе решения. Это LibreChat, BGE-M3 и Qwen2.5-0.5B на vLLM. Для Indexer и Retriever вопрос остаётся открытым.

Для сравнения конфигураций Indexer и Retriever построен единый пайплайн с подменяемыми компонентами. Качество аналитики оценивается по четырём критериям LLM-судьи — Comprehensiveness, Empowerment, Diversity и Overall — по шкале от 1 до 10. Для воспроизводимости оценок судья запускается 5 раз при temperature=0, итоговый балл вычисляется как среднее. Вычислительная стоимость индексации измеряется как throughput (токен/сек).

Как видно из таблицы, [результаты]. Вопрос остаётся в том, какая конфигурация обеспечивает качество аналитики не ниже порога α при сниженной вычислительной стоимости индексации.

Поэтому рассматривается конфигурация, в которой вместо LLM для извлечения сущностей используется spaCy NER+RE, вместо LLM для суммаризации кластеров используются centroid-эмбеддинги.

## Слайд 4: Цели и задачи

Целью работы является проектирование и реализация компонентов RAG-инфраструктуры для аналитики корпоративных документов в облачных средах на основе модификации LeanRAG с устранением LLM-вызовов на этапе индексации. Полученная конфигурация должна обеспечить качество аналитики не ниже порога α при сниженной вычислительной стоимости индексации.

Научная новизна работы состоит в том, что она является первым исследованием зависимости retrieval-качества LeanRAG от наличия LLM при индексации и предлагает верифицированную модификацию, устраняющую LLM-вызовы при индексации без потери качества ниже порога α.

Поставлены четыре задачи.

Задача 1. Реализовать базовый прототип RAG-инфраструктуры. Query Processor на основе LibreChat, Organizer на основе BGE-M3, Generator на основе инференса Qwen2.5-0.5B на vLLM, Retriever и Indexer на основе flat vector search. Этот прототип является baseline для сравнения.

Задача 2. Внедрить LeanRAG в качестве Retriever и Indexer, заменив плоский поиск на граф знаний с иерархическим retrieval.

Задача 3. Модифицировать LeanRAG, заменив LLM-извлечение сущностей на spaCy NER+RE и LLM-суммаризацию кластеров на centroid-эмбеддинги.

Задача 4. Провести сравнительную оценку по метрикам Comprehensiveness, Empowerment, Diversity, Overall, recall@k и latency.

## Слайд 5: Объект, предмет, методы

Объектом исследования является облачный сервис для работы с корпоративными документами в форматах PDF, DOCX и CSV.

Предметом исследования являются компоненты RAG-инфраструктуры на базе LeanRAG. Это Indexer, включающий NER/RE, граф знаний и GMM-иерархию, и Retriever с bottom-up hybrid retrieval.

Методы исследования включают четыре этапа. Этап 1. Аналитический обзор GraphRAG, LightRAG и LeanRAG с выявлением gap в качестве и стоимости индексации. Этап 2. Анализ кода LeanRAG с локализацией LLM-вызовов при построении иерархии. Этап 3. Реализация модификации в Python с заменой LLM на spaCy NER+RE и centroid-эмбеддинги. Этап 4. Сравнительное тестирование прототипа и baseline по метрикам Comprehensiveness, Empowerment, Diversity, Overall, recall@k и latency.

## Слайд 6: Постановка задачи прототипа

Третий метод предполагает реализацию прототипа. Посмотрим, что именно он должен делать.

Документы поступают в Indexer. Indexer строит граф знаний через spaCy NER+RE без LLM и формирует GMM-иерархию уровней через centroid-эмбеддинги без LLM. Параллельно строится векторный индекс BGE-M3.

Запрос пользователя поступает в Retriever. Retriever выполняет bottom-up traversal по иерархии и vector search fallback через BGE-M3. Извлечённый контекст передаётся в Organizer, а затем в Generator.

Требование к прототипу одно. Он должен превысить baseline по всем метрикам качества ответа при latency не более 5 секунд.

---

## Слайд 7: Архитектура решения

Перейдём к реализации архитектуры.

Зелёным выделены мои модификации, синим выделены компоненты LeanRAG без изменений и сливовым другие компоненты.

Со стороны инициализации, документы поступают в Indexer. Chunker разбивает документы на чанки. spaCy NER+RE извлекает сущности и связи без LLM. Сущности формируют Knowledge Graph, чанки эмбеддируются через BGE-M3 в Vector Store. GMM Clustering строит иерархию уровней через centroid-эмбеддинги.

Со стороны запроса, клиент МТС отправляет запрос в LibreChat и поступает в Retriever. Retriever выполняет bottom-up traversal по иерархии Knowledge Graph и vector search fallback по Vector Store. Контекст из обоих каналов объединяется в Organizer и передаётся в Generator на vLLM.

## Слайд 8: План экспериментальной оценки

Для оценки прототипа собран датасет из корпоративных документов в форматах PDF, DOCX и CSV. К документам составлены синтетические вопросы с эталонными ответами. Якоря в виде чисел, дат и сумм размечены вручную для валидации faithfulness. В качестве baseline взято текущее состояние МТС. Это LibreChat с BGE-M3 и flat vector retrieval.

LLM-судья оценивает Comprehensiveness, Empowerment, Diversity и Overall по шкале от 1 до 10. Дополнительно измеряются faithfulness, groundedness, recall@k и throughput индексации в токенах в секунду.

Для воспроизводимости судья запускается 5 раз при temperature=0, итоговый балл вычисляется как среднее по прогонам.

---

## Слайд 9: Ожидаемый практический эффект

Ожидаемые технические улучшения относительно baseline.

Recall@k должен вырасти, так как иерархия связывает сущности из разных документов, которые flat search теряет.

Comprehensiveness, Empowerment, Diversity и Overall должны вырасти за счёт компактного retrieved context. По данным статьи LeanRAG, retrieved context составляет 0.9 миллиона токенов на датасете Mix, что значительно меньше, чем у конкурентов.

LLM-вызовы при индексации устранены заменой NER+RE на spaCy и суммаризации кластеров на centroid-эмбеддинги.

Latency не более 5 секунд.

Прикладной эффект. Объективные метрики качества вместо субъективной удовлетворённости клиентов. Прозрачный retrieval с видимостью того, из каких сущностей и кластеров собран контекст. Прототип совместим с LibreChat как drop-in замена Retriever.

## Слайд 10: Заключение

Выявлен gap. LeanRAG даёт лучшее качество и наименьший retrieved context среди рассмотренных методов, но LLM вызывается на каждом уровне иерархии при индексации.

Предложена модификация LeanRAG. spaCy NER+RE вместо LLM-извлечения сущностей и centroid-эмбеддинги вместо LLM-суммаризации кластеров.

Иерархия и bottom-up retrieval LeanRAG сохранены без изменений.

Определены метрики и baseline. Comprehensiveness, Empowerment, Diversity, Overall, recall@k и latency оцениваются относительно flat retrieval.

Следующие шаги. Реализация модификации LeanRAG и экспериментальная оценка относительно baseline.

Спасибо за внимание.

## Слайд 11: Реальный кейс — МТС

В МТС работает LibreChat с BGE-M3 и flat vector retrieval без иерархии. Клиенты загружают PDF, DOCX и CSV и задают вопросы на естественном языке. Метрики не измеряются, качество оценивается через субъективную удовлетворённость клиентов.

Плоский retriever теряет связи между сущностями из разных документов и не даёт видимости в то, что именно извлекается.

Прототип даёт три результата. Результат 1. Drop-in замена Retriever, совместимая с LibreChat без изменения интерфейса. Результат 2. Объективные метрики recall@k и latency вместо субъективного мнения. Результат 3. Иерархический граф для ответов на вопросы, требующих синтеза из нескольких документов.

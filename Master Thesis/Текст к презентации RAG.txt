Слайд 2: Актуальность
================================================================================
Начну с актуальности проблемы. Современные облачные системы генерируют терабайты логов, метрик и трассировок ежесуточно. Ручной анализ инцидентов физически не успевает за таким объёмом данных, что приводит к росту времени обнаружения и устранения сбоев — метрик MTTD и MTTR — и, как следствие, к снижению SLA.

Большие языковые модели умеют анализировать неструктурированный текст, но они ограничены размером контекста — обычно от 8 до 128 тысяч токенов. Retrieval-Augmented Generation, или RAG, решает эту проблему: система сначала извлекает только релевантные данные из большой базы знаний, а затем подаёт их LLM для генерации ответа.

Однако здесь возникает новая проблема: существующие RAG-системы либо слишком просты — используют плоский векторный поиск и теряют связи между сущностями — либо слишком дороги, потому что строят иерархии графов знаний через LLM-суммаризации на каждом уровне, что требует миллионов токенов.

Кроме того, важна прозрачность анализа. Техника Chain of Thought позволяет LLM раскрывать ход рассуждений пошагово: наблюдение, гипотеза, проверка, решение. Это делает выводы понятными для SRE-инженеров.

Таким образом, нужна оптимизация двух компонентов: эффективный граф знаний для извлечения контекста и прозрачное CoT reasoning для анализа.

================================================================================
Слайд 3: Обзор существующих решений
================================================================================
Давайте посмотрим на существующие решения и их ограничения.

Доменные агенты, такие как K8sGPT и Firefighting-GPT, работают "из коробки" с Kubernetes. Они используют простой prompt-engineering без глубокого извлечения знаний, поэтому их точность резко падает на нетривиальных инцидентах.

Универсальные фреймворки — LangChain, PromptFlow, Semantic Kernel — отлично оркестрируют вызовы LLM и позволяют строить CoT-агенты, но они не решают проблему эффективного извлечения контекста из больших графов знаний.

Теперь о RAG-фреймворках. GraphRAG и HippoRAG действительно строят иерархии графов знаний, но для этого на каждом уровне иерархии они генерируют LLM-суммаризации. На больших базах знаний — например, больше миллиона узлов — это требует миллионов токенов и становится слишком дорого.

С другой стороны, плоские RAG-системы выполняют обычный top-k поиск по векторным эмбеддингам без учёта структуры знаний. Они теряют связи между сущностями — например, связь между ошибкой в логах и метрикой потребления памяти.

Особый интерес представляет LightRAG. По документации он обещает граф-ориентированный подход с иерархическим обходом компонент связности. Однако, когда я проанализировал исходный код, оказалось, что реальная имплементация ищет top-k entities по векторной близости, затем все инцидентные рёбра — и всё. Реального обхода по слоям нет. Но LightRAG почти подходит для нашей задачи — он имеет правильную архитектуру, просто требует оптимизации для повышения точности retrieval.

Что касается inference-движков для LLM, я выбрал vLLM — это лучший из существующих движков по производительности и эффективности. Он станет backend для нашего CoT reasoning.

один пробел для этих решений — нет гарантированной PII-маскировки, нет прозрачного пошагового объяснения, и точность падает, когда данных слишком много или инцидент нетривиален.

Таким образом, я выбрал LightRAG с оптимизациями для retrieval, vLLM для inference, и LangChain для построения CoT-агента.

Слайд 4: Цели и задачи
================================================================================
Исходя из этого, я поставил цель работы: спроектировать RAG-систему для анализа инцидентов в облачной инфраструктуре MTS на основе оптимизированного LightRAG и vLLM inference.

Для достижения этой цели я сформулировал четыре задачи:

Первая задача — спроектировать архитектуру системы. Определить, какие компоненты необходимы для полного цикла обработки инцидента: от сбора телеметрии до генерации отчётов. Как эти компоненты взаимодействуют друг с другом? Какие данные передаются между ними?

Вторая задача — разработать оптимизации для LightRAG. Это ключевой пункт работы. Базовая имплементация LightRAG использует плоский 1-hop retrieval без реальных иерархий. Мне нужно спроектировать три оптимизации: hierarchical graph traversal для настоящего обхода по слоям графа, cumulative embeddings вместо дорогостоящих LLM-суммаризаций на каждом уровне иерархии, и adaptive pruning для сокращения search space и ускорения retrieval.

Третья задача — разработать методику оценки. Определить технические метрики: retrieval precision для оценки качества извлечения, latency для оценки скорости, memory consumption для оценки эффективности использования ресурсов. Также определить бизнес-метрики: MTTD и MTTR, которые важны для SRE-команды. И выбрать baseline для сравнения: K8sGPT, flat vector search RAG, и ручной анализ SRE.

Четвёртая задача — провести реализацию и экспериментальную валидацию системы на production данных MTS. Это включает реализацию оптимизированного retrieval layer, построение knowledge graph на реальных данных, интеграцию с CoT agent, проведение экспериментов на 50 инцидентах, и валидацию с SRE-командой МТС.

Слайд 6: Постановка задачи прототипа
================================================================================
Теперь давайте детально разберём постановку задачи для прототипа.

Работа начинается, когда система получает alert об инциденте — например, из Prometheus Alertmanager. Из этого alert извлекается query — описание проблемы.

Далее запускается LightRAG retrieval: система выполняет иерархический обход графа знаний, который содержит логи, метрики и документацию Kubernetes. На выходе получаем top-N релевантных ченков с учётом connectivity — то есть связей между сущностями в графе.

Следующий шаг — Context Management: полученные chunks ранжируются по relevance и recency, проходят PII-маскировку для безопасности, сжимаются, чтобы убрать дубликаты, и укладываются в token limit языковой модели.

Затем подготовленный контекст передаётся CoT Agent, который построен на LangChain с vLLM в качестве inference backend. Агент выполняет пошаговый анализ: 
- observation — что произошло? 
- hypothesis — какие возможные причины? 
- verification — проверка гипотез по метрикам и логам
- conclusion — root cause и конкретные рекомендации.

Система генерирует отчёт за не более 60 секунд с прозрачным reasoning. Если уверенность агента превышает порог 0.8, автоматически создаётся тикет в Jira.

================================================================================
Слайд 7: Архитектура решения
================================================================================
Архитектура решения состоит из пяти компонентов
Первый компонент — Сбор и индексация. Данные из трёх источников — Prometheus для метрик, Loki для логов и документация Kubernetes — проходят обработку. Система извлекает entities, то есть сущности типа pod, service, node; relations — связи между ними; и chunks — фрагменты текста. Из всего этого строится knowledge graph с иерархическими слоями. Ключевой момент: иерархии строятся через cumulative embeddings — агрегацию векторов — что позволяет избежать дорогостоящих LLM summaries на каждом уровне. Это снижает token overhead на 70–80 процентов.

Второй компонент — Optimized Retrieval Layer на базе доработанного LightRAG. Когда поступает query от alert об инциденте, система выполняет векторный поиск top-k entities на верхнем уровне иерархии. Затем начинается спуск по графу с использованием connectivity signals — это метрики связности узлов, которые помогают понять, какие части графа наиболее релевантны. Применяется adaptive pruning для ограничения search space — мы не обходим весь граф, а умно сужаем область поиска. В итоге извлекается финальный набор chunks, которые действительно релевантны запросу.

Третий компонент — Context Management. Это критически важный этап подготовки данных для LLM. Извлечённые chunks ранжируются по двум параметрам: relevance — насколько они релевантны запросу, и recency — насколько свежие данные. Затем они проходят compression для удаления дубликатов и избыточной информации. И наконец, система гарантирует укладку в token limit LLM — обычно от 8 до 32 тысяч токенов в зависимости от модели.

Теперь перейдём к нижнему ряду — это анализ и вывод результатов.

Четвёртый компонент — CoT Agent на базе LangChain и vLLM. Это сердце системы для анализа. Агент выполняет пошаговый анализ по схеме Chain of Thought: 
Сначала observation — что произошло? Агент анализирует симптомы из логов и метрик.
Затем hypothesis — какие возможные причины? Формируется несколько гипотез на основе patterns.
Далее verification — проверка гипотез. Агент ищет в логах и метриках подтверждения или опровержения каждой гипотезы.
И наконец conclusion — итоговый вывод с указанием root cause и конкретными рекомендациями по устранению.

vLLM здесь играет ключевую роль как inference backend. Это лучший из существующих движков для LLM inference по производительности, что позволяет обрабатывать запросы быстро и эффективно.

Пятый компонент — Output. Система формирует финальный отчёт с прозрачным reasoning — все шаги CoT видны SRE-инженеру — и отправляет его в Slack. Если уверенность системы, которая вычисляется по специальной формуле с учётом качества retrieval, согласованности reasoning и наличия evidence, превышает порог 0.8, то автоматически создаётся тикет в Jira с описанием проблемы и рекомендациями.

Кроме того, все подтверждённые кейсы — то есть инциденты, где рекомендации системы оказались правильными — сохраняются в векторное хранилище. Эти данные потом используются для fine-tuning retrieval модели, что замыкает цикл обучения системы на реальных данных.

Слайд 8: План экспериментальной оценки
================================================================================
Теперь о плане экспериментальной оценки.

Для экспериментов я подготовил датасет из 50 синтетических инцидентов на исторических логах Prometheus и Loki из production среды MTS. Синтетические — в том смысле, что я воспроизвожу реальные типы инцидентов на исторических данных, чтобы можно было контролируемо тестировать.

Для сравнения я использую три baseline: первый — K8sGPT с prompt-engineering; второй — flat vector search RAG без графа; и третий — ручной анализ SRE-команды, чтобы понять, насколько система приближается к экспертному уровню.

Я буду измерять технические метрики: 
Retrieval precision@10 и recall@10 для оценки качества извлечённых chunks.
End-to-end latency — время от получения alert до генерации отчёта, целевое значение не более 60 секунд.
Memory consumption при построении иерархий — сравню cumulative embeddings с LLM summaries.

Также буду измерять бизнес-метрики: MTTD — Mean Time To Detect, и MTTR — Mean Time To Resolve. Сравню команду без агента и с агентом CoT-LLM.

Кроме того, проведу экспертную оценку: опрошу SRE-инженеров о понятности CoT reasoning через transparency score, о доверии к рекомендациям через confidence score, и об actionability — то есть можно ли рекомендации сразу применить на практике.

Эксперименты будут проведены после завершения реализации системы. На диаграмме показаны целевые значения метрик, которые система должна достичь для подтверждения эффективности предложенного подхода.

Слайд 9: Ожидаемый практический эффект
================================================================================
Перейдём к ожидаемым результатам. Я разделил их на технические улучшения и бизнес-эффект.

Среди технических улучшений:

Во-первых, я ожидаю retrieval precision@10 не менее 0.85 по сравнению с baseline flat search около 0.65. Это значит, что иерархический обход графа будет точнее находить релевантные данные.

Во-вторых, memory efficiency: построение иерархий через cumulative embeddings без LLM summaries должно снизить token overhead на 70–80 процентов.

В-третьих, end-to-end latency не более 60 секунд для типового инцидента — это включает retrieval, CoT reasoning и inference.

В-четвёртых, CoT transparency: я ожидаю, что 95 процентов шагов reasoning будут понятны SRE без дополнительных объяснений. Это буду измерять через опрос.

Что касается бизнес-эффекта:

Во-первых, MTTD должен снизиться на 30–40 процентов, потому что автоматическое извлечение релевантных данных из графа знаний ускоряет обнаружение проблемы.

Во-вторых, MTTR должен снизиться на 20–40 процентов благодаря тому, что CoT reasoning выдаёт actionable recommendations с прозрачным обоснованием.

В-третьих, экономия около двух часов на SRE-инженера в сутки за счёт автоматизации рутинного триажа.

И наконец, выравнивание экспертизы: junior SRE будут получать рекомендации senior-уровня благодаря прозрачному пошаговому анализу системы.

Все эти гипотезы будут проверены после реализации системы и проведения экспериментальной валидации.

# Annotator Guide

## Intuition
An annotator is the only agent with access to ground truth. Every decision made here propagates into all downstream metrics — errors at this stage cannot be corrected later.

## Definition

An annotator produces a set of triples $(q_i, a_i^*, A_i)$ for [[Eval Dataset]] where $A_i$ are verbatim anchor strings extracted directly from source documents. No RAG system may run before anchors are extracted — see [[Evaluation Agents]] Property 1.

## Step-by-step instructions

### Before you start

Read every document completely before writing a single question. Annotations written after partial reading bias toward salient passages at the expense of buried facts.

Have the document open in a plain text viewer, not through any RAG interface.

### Question density

| Document length | Minimum questions | Distribution |
|-----------------|------------------|--------------|
| < 800 tokens | 5 | 3 factoid + 1 open-ended + 1 paraphrase |
| 800–3 000 tokens | 8–10 | 4 factoid + 2 open-ended + 2 paraphrase + 1 multi-hop |
| > 3 000 tokens | 12–15 | cover all [[Question Taxonomy]] cells |

For cross-document multi-hop: write at least 3 questions per document pair that share an entity.

**Density rationale.** RAGBench (2024) observes that fewer than 5 questions per document leave entire sections untested. GaRAGe (ACL 2025) uses ~15 passages per question; for a thesis corpus the inverse ratio of 5–10 questions per document is appropriate.

### Writing a question

A valid question satisfies all four conditions:
1. **Answerable** — the answer exists verbatim or by direct inference from the document
2. **Non-trivial** — cannot be answered from the question itself or general knowledge alone
3. **Grounded** — at least one verbatim substring from the document supports the answer
4. **Typed** — explicitly assigned a cell in [[Question Taxonomy]]

### Extracting anchors

An anchor is a verbatim substring of the source document, not a paraphrase.

```
CORRECT anchor: "общая стоимость услуг составляет 4 500 000 (четыре миллиона пятьсот тысяч) рублей"
WRONG anchor:   "стоимость договора — 4,5 млн рублей"   ← paraphrase, not verbatim
```

Extract the shortest substring that uniquely identifies the answer. Multiple anchors per question are allowed and recommended for multi-hop.

### Taxonomy tagging

For every question, record:

| Field | Value |
|-------|-------|
| `factuality` | `factoid` or `open-ended` |
| `premise` | `direct` or `with-premise` |
| `complexity` | `single-hop` or `multi-hop` |
| `terminology` | `in-document` or `paraphrase` |
| `source_doc` | filename(s) |

### Writing your score (for [[Human Annotation]])

For each system answer to your question, score on four dimensions (1–10):

| Dimension | Meaning |
|-----------|---------|
| Comprehensiveness | Does the answer cover all aspects of the question? |
| Empowerment | Does the answer help make an informed judgment? |
| Diversity | Does the answer avoid repetition and consider multiple perspectives? |
| Overall | Holistic quality |

Score independently before reading other systems' answers. Score all systems for one question before moving to the next (within-question anchoring reduces between-question drift).

### Anti-bias checklist

Before submitting annotations:
- [ ] All anchors were extracted before any RAG system ran
- [ ] No question was generated by looking at a system's answer
- [ ] At least 2 taxonomy dimensions vary across the question set
- [ ] Each document has at least 5 questions
- [ ] At least 3 cross-document multi-hop questions exist

## Output format

Each row in `datasets/queries.jsonl`:

```json
{
  "id": 1,
  "query": "Какова общая стоимость услуг по договору?",
  "expected_answer": "4 500 000 рублей",
  "anchors": ["общая стоимость услуг составляет 4 500 000"],
  "type": {"factuality": "factoid", "premise": "direct", "complexity": "single-hop", "terminology": "in-document"},
  "source_doc": ["contract_001.txt"]
}
```

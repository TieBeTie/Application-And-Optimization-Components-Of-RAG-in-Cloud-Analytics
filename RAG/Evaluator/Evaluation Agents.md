# Evaluation Agents

## Intuition
A RAG evaluation involves three distinct agents. Conflating any two of them introduces systematic bias that inflates scores independently of real system quality.

## Definition
An evaluation of a RAG system involves three agents:

| Agent | Input | Output |
|-------|-------|--------|
| **Annotator** | source documents | queries $q_i$, anchors $A_i$, expected answers $a_i^*$ |
| **Generator** | query $q_i$, retrieved chunks | answer $\hat{a}_i$ |
| **Judge** | query $q_i$, answer $\hat{a}_i$ | score $s_i \in [1, 10]$ |

## Properties

### Property 1 — Annotator ⊥ Generator
The Annotator must not observe Generator outputs during annotation.

If the Annotator sees Generator outputs, queries drift toward what the Generator answers well — the dataset becomes a self-fulfilling benchmark.

### Property 2 — Judge ⊥ Generator
The Judge must be a different model family than the Generator.

If Judge and Generator share the same prior, the Judge assigns higher scores to outputs stylistically similar to its own generation, independent of factual correctness.

### Property 3 — Annotator ⊥ Generator model family
Synthetic queries must be generated from raw document text, not from Generator responses.

If synthetic queries are generated by querying the Generator, they are biased toward facts the Generator represents well in its internal structure (e.g., graph nodes in LeanRAG, dense clusters in GraphRAG).

### Examples
**Violation of Property 1.** Human annotator runs LightRAG, reads its answers, then writes questions. Questions cover exactly what LightRAG retrieved — hard questions for LightRAG disappear from the dataset.

**Violation of Property 2.** GPT-4o is used as Generator and GPT-4o as Judge. Judge scores GPT-4o outputs higher due to shared stylistic prior.

**Violation of Property 3.** Q&A pairs generated by querying LightRAG, then used to evaluate LightRAG — LightRAG's graph structure determines which facts appear as questions.
